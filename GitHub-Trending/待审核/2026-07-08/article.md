# 今日 GitHub AI 热点：三个项目分别做求职、办公和本地语音

> 今天日榜前列以"给 AI 智能体加一层具体能力"为主线：一个把 Claude Code 变成求职助手，一个让智能体直接读写 Office 文件，一个把语音合成压进 CPU。三个项目方向各异，但都指向同一件事——把大模型接到一个能落地的工作流里。

---

## 1. ai-job-search — 建在 Claude Code 上的求职申请框架

| 总 Star | 24h 新增 | 主语言 | 协议 |
|---|---|---|---|
| 12k | 2.5k | TypeScript | MIT |

ai-job-search 解决的问题很具体：把求职这件事拆成"填资料、搜岗位、评估匹配、改简历、写求职信、准备面试"几步，再让 Claude Code 按流程跑下来。用户 fork 仓库、填好个人档案后，用 `/setup`、`/scrape`、`/apply <url>` 三个命令就能走完从搜岗到出简历的链路。

它的做法不是让模型一次性生成简历，而是搭了一条"起草—评审"流水线：一个 agent 起草 CV 和求职信，另一个 agent 按结构化标准做批评和修订，简历用 LaTeX 编译，还带一个 ATS 可解析性检查。核心评估流程是语言和国家无关的，只有岗位搜索技能目前对接的是丹麦本地招聘网站，README 里说明了替换成其他地区岗位板的方式。

适合想用 Claude Code 管理求职流程、又愿意自己改配置的开发者参考。项目创建于 2026 年 3 月，目前已有一定规模。

🔗 项目地址：https://github.com/MadsLorentzen/ai-job-search

## 2. OfficeCLI — 让 AI 智能体读写 Office 文件的命令行工具

| 总 Star | 24h 新增 | 主语言 | 协议 |
|---|---|---|---|
| 10.5k | 893 | C# | Apache-2.0 |

OfficeCLI 面向的是 AI 智能体处理 Word、Excel、PowerPoint 的场景。它是一个单二进制、不依赖本机安装 Office 的工具，让智能体通过命令行去读取、编辑和自动生成三类文档。

它的差异点在内置的 HTML 渲染引擎：把 `.docx` / `.xlsx` / `.pptx` 渲染成 HTML 或 PNG，让智能体能"看到"文档排版，从而形成"渲染—检查—修正"的闭环，而不是盲改 XML。接入方式也偏工程化，一条 `officecli install` 会把技能文件装进它能识别的多个编码智能体里，包括 Claude Code、Cursor、Windsurf 等，无需额外配置。

适合做文档自动化、需要智能体批量出报表或演示文稿的团队参考。项目采用 Apache-2.0 协议，创建于 2026 年 3 月。

🔗 项目地址：https://github.com/iOfficeAI/OfficeCLI

## 3. pocket-tts — 能跑在 CPU 上的轻量语音合成模型

| 总 Star | 24h 新增 | 主语言 | 协议 |
|---|---|---|---|
| 6.3k | 531 | Python | MIT |

pocket-tts 来自 Kyutai 实验室，目标是让文本转语音不再依赖 GPU 和云端 API：pip 装好、调一个函数就能出音频。模型只有 1 亿参数，README 给出的数据是首个音频块延迟约 200ms，在 MacBook Air M4 上约为实时速度的 6 倍，且只占用 2 个 CPU 核心。

功能上它支持音频流式输出、声音克隆，覆盖英语、法语、德语、葡萄牙语、意大利语、西班牙语六种语言，并能处理任意长度的文本输入，还提供了浏览器端运行的实现。项目挂出了 arXiv 论文和 Hugging Face 模型卡，技术资料相对完整。

适合需要在本地或边缘设备做语音合成、对延迟和隐私敏感的场景参考。相比要配置 GPU 的方案，它把部署门槛压到了一次 pip 安装。

🔗 项目地址：https://github.com/kyutai-labs/pocket-tts

## 今日观察

今天前列三个项目的共同点，是都在做"大模型的最后一公里"——不是训练新模型，而是给已有模型接一个能干活的接口：接到求职流程、接到 Office 文件、接到本地音频输出。相比追逐更大的模型，这类把能力落到具体工作流的项目，最近在榜单上出现得更频繁。
