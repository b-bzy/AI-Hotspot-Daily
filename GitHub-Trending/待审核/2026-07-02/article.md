# 今日 GitHub AI 热点：Agent 工具链从运行到隔离逐渐成型

> 今天日榜上 AI 相关的新面孔集中在两端：一端是给大模型准备训练数据的文档处理工具，另一端是围绕编码 Agent 的运行与隔离基础设施。三个项目分别对应数据、编排、沙箱三个环节，凑在一起正好能看出 Agent 工具链正在补齐的几块。

---

## 1. olmOCR — 把 PDF 转成干净文本、供大模型训练用的工具包

| 总 Star | 24h 新增 | 主语言 | 协议 |
|---|---|---|---|
| 18.4k | 334 | Python | Apache-2.0 |

olmOCR 由 Allen Institute for AI（Ai2）维护，解决的问题是把 PDF、PNG、JPEG 这类图像化文档转成干净的 Markdown 纯文本。它能处理公式、表格、手写内容和多栏排版，会自动去掉页眉页脚，并按自然阅读顺序输出，主要面向"给大模型准备训练语料"这个场景。

技术上它基于一个 7B 参数的视觉语言模型（VLM），因此需要 GPU 运行。README 给出的成本数据是每转换一百万页低于 200 美元。项目还配套了 olmOCR-Bench 评测集，覆盖 1,400 份文档、7,000 多个测试用例，用来横向对比不同 OCR 系统。最新的 v0.4.0 引入了 RL 训练，官方称基准分数相比前一版提升约 4 分。

适合需要批量清洗文档、构建 LLM 数据集的团队，以及做 OCR 效果对比评测的研究场景。

🔗 项目地址：https://github.com/allenai/olmocr

## 2. herdr — 在终端里同时管理多个编码 Agent 的多路复用工具

| 总 Star | 24h 新增 | 主语言 | 协议 |
|---|---|---|---|
| 9.7k | 609 | Rust | 自定义（见仓库） |

herdr 面向"同时跑好几个编码 Agent"的开发者。它给每个 Agent 分配一个真实终端而不是模拟窗口，侧边栏把每个 Agent 的状态汇总成阻塞、工作中、完成、空闲四种，让用户一眼看出哪个需要介入。作者的定位是"为 Agent 时代重写的 tmux"。

和 tmux 相比，herdr 的差异点在于内建了 Agent 状态感知，不需要自己去接铃声或钩子；和 conductor、cmux 这类图形化管理器相比，它是一个约 10MB 的单一 Rust 二进制文件，直接跑在现有终端里，能通过 SSH 从服务器甚至手机重新接入，断开连接后进程不会中止。它还提供本地 socket API，Agent 可以通过它自行编排。

适合需要并行调度多个编码 Agent、且习惯终端工作流的开发者。项目创建于 2026 年 3 月，目前 Windows 版本仍为测试阶段。

🔗 项目地址：https://github.com/ogulcancelik/herdr

## 3. CubeSandbox — 为 AI Agent 提供硬件级隔离的轻量沙箱

| 总 Star | 24h 新增 | 主语言 | 协议 |
|---|---|---|---|
| 6.9k | 79 | Rust | Apache-2.0 |

CubeSandbox 由腾讯云开源，用于给 AI Agent 执行代码提供一个安全隔离的运行环境。它基于 RustVMM 和 KVM 构建，README 给出的数据是在 60 毫秒内创建一个硬件隔离的沙箱，内存开销低于 5MB，同时兼容 E2B SDK，既能单机部署也能扩展到多节点集群。

在功能迭代上，v0.3.0 引入了名为 CubeCoW 的写时复制快照引擎，支持事件级快照、即时克隆和回滚到任意保存状态；最新的 v0.4.0 增加了凭证保管库，让 Agent 调用外部 API 时密钥不进入沙箱内部。项目已被收录进 CNCF Landscape 的 AI-native 基础设施分类。

适合需要大规模、高密度运行不可信 Agent 代码，并对隔离和启动速度有要求的平台方。

🔗 项目地址：https://github.com/TencentCloud/CubeSandbox

## 今日观察

三个项目分处 Agent 工具链的不同环节：olmOCR 负责把文档变成可训练的数据，herdr 负责在终端里编排多个 Agent，CubeSandbox 负责给 Agent 的代码执行加一层隔离。它们之间没有直接关联，但放在一起能看出，围绕 Agent 的配套工程正在从模型本身，向数据、运行、安全等外围环节展开。
