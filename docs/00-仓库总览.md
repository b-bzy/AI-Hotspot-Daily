## 五、可选的物理重构方案(需要你拍板,暂未执行)

理想结构是把日报模块也收进自己的文件夹(和另两个模块对称):

```
日报/(SKILL.md + 配置.json + reference/ + 待审核/)   ← 根目录只留 README/docs/scripts/.github
共享/(爆款选题评分框架.md + 串烧写作规范.md)          ← 跨模块共享的规范单独放
```

**为什么没直接做**:这些路径被 **4 个云端 routine 的 prompt、5 个 GitHub Action 的路径触发器、2 个脚本的相对路径、3 个本地任务的文档**同时引用。挪动=同步改 14 处引用,任何一处漏改,第二天早上对应流水线就断。风险收益比不划算——**当前结构的问题靠 docs/ 文档层已经解决了"找不到东西"**,物理不对称只是强迫症层面的不完美。

如果你仍想做,我可以执行,但建议:①挑一个没有内容产出的时间窗(如晚上);②我一次性改完 14 处引用并逐条 routine 手动触发验证;③预留一天观察期。你说"执行物理重构"我就按这个来。

## 六、排障速查

| 症状 | 先看哪 |
|---|---|
| TG 没收到某条推送 | GitHub 仓库 Actions 页对应工作流的日志 → 再查 routine 是否推了 commit |
| 日报数据缺某个源 | `待审核/日期/fetch_report.json` 的失败记录 + `reference/sources.md` 的坑备注 |
| 云端 routine 没跑/失败 | claude.ai → Code → 对应 routine 的会话日志 |

| 想改推送时间/内容 | routine 的 cron 和 prompt(找 Claude 改);Action 只管"推上来就发" |
