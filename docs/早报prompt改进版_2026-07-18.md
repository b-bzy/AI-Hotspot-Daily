# AI 早报 prompt · 改进版（防漏语音/Sales）· 2026-07-18

> 目标：降低语音、Sales 两个板块每日"漏抓"和"误报 🈚️"的概率。
> 改动范围：**只改第 2 步（信息源策略）和第 3 步（板块 A/B 定义 + 兜底规则）、第 8 步报告加一行**。
> 分支规则、1.5 去重、第 4-7 步（日期/模板/HTML 约束/推送）**一字未动**——生产环境已验证的部分不碰。

---

## 一、为什么会漏（诊断）

| 病根 | 后果 | 对应改法 |
|---|---|---|
| **100% 公司名驱动搜索** | 名单外的新公司永远发现不了。实测：本周语音最重要两条来自阶跃星辰，旧名单没有它 → 整块漏掉 | 新增 **主题驱动搜索**（按赛道关键词搜，不依赖公司名） |
| **11/22 个 X handle 是错的** | `site:x.com @错handle` 每天都是空枪，白烧搜索预算 | 全部替换为已核准 handle |
| **名单含已死条目** | Drift（已关停）、Air AI（被 FTC 禁）每天白搜 | 剔除 |
| **X 搜不到就没有回退** | WebSearch 对 X 索引很差，搜不到≠没发生 | 新增 **回退路径**：官网 newsroom / PR 稿 / 媒体 |
| **无会议日历意识** | 大厂发布高度集中在 WAIC/CES/Build/I/O/Dreamforce 等窗口，撞上就是集中漏 | 新增 **会议窗口检查** |
| **🈚️ 兜底门槛太低** | "搜了一下没有"就宣布 0 条，实际是搜法不对 | 新增 **穷尽检查清单**：跑完规定搜索模式才准宣布 🈚️ |
| **A/B 边界含糊** | 销售外呼 Voice Agent（Bland/Vapi）算语音还是 Sales？可能两边都不收 | 明确边界规则 + "拿不准往 A/B 放"原则 |

## 三、改进后的第 3 步（板块 A / B 定义 + 兜底规则）

```
   ━━━ 板块 A：语音模型动态（公司业务相关，最高优先级）━━━
   - 全球语音模型（TTS / ASR / Voice Cloning / 实时对话语音 / Voice Agent / 语音 LLM /
     语音芯片 / AI 音乐音频生成）的最新发布、研究成果、行业动态
   - 关注厂商见 2.1；**主题搜索捞到的名单外公司同样收**
   - 论文 / 技术博客 / 产品发布 / 融资 / 收购 / 合作 / 政策 / 榜单变动 / X 官号公告 都算
   - 语音相关的**硬件**（AI 眼镜、AI 耳机、翻译机、智能体手机的语音入口）也算 A

   ━━━ 板块 B：AI for Sales 动态（公司业务相关，最高优先级）━━━
   - AI 在销售场景的应用：AI SDR / BDR、销售外呼 Agent、销售线索生成、CRM AI、
     对话智能 / 录音分析、销售辅助 Copilot、销售流程自动化、revenue intelligence
   - 关注厂商见 2.1（含中国厂商）；**主题搜索捞到的名单外公司同样收**
   - 产品发布 / 融资 / 收购 / 客户案例 / 研究报告 / 财报相关数据 / X 官号公告 都算

   ⚠️ A / B 边界规则（避免两边都不收）：
   - **语音模型、语音基建本身** → 板块 A（例：Cartesia 发新 TTS、LiveKit 融资）
   - **语音用在销售场景** → 板块 B（例：Bland 做 AI 外呼、Vapi 拿下客服呼入）
   - 同一家公司可能两边都出现，只要是两件不同的事，不算重复
   - **拿不准归 A 还是 B 时，选一个放进去，绝不要因为犹豫而丢弃**——A、B 是业务板块，
     宁可放宽也不要漏；只有确定与语音、销售都无关时才放 C

   ⚠️ 三个板块统一规则：
   - 没有条数下限，也没有条数上限
   - 24h 内实际找到多少条优质 + 去重后的内容，就写多少条
   - 不要为了"看起来丰富"硬凑边角料新闻；宁可少而精

   ⚠️ 宣布 🈚️ 之前的穷尽检查（**新增，针对 A、B 两个板块**）：
   板块 A 或 B 想写"🈚️ 过去 24 小时内无显著进展"之前，必须确认已经跑完：
     ① 该板块 2.2 的全部主题驱动搜索（至少 3 条）
     ② 该板块名单里至少 8 家头部公司的公司名搜索
     ③ 至少 1 条中文搜索
     ④ 若今天落在 2.5 的会议窗口内，还要跑完该会议的专项搜索
   四项都跑完仍无结果，才允许写 🈚️，并在第 8 步报告里注明"已跑完穷尽检查"。
   （C 板块不受此约束，维持原规则。）
```

## 四、第 8 步报告加两行

```

   - 板块 A 实际找到 X 条（列出标题，0 条则注明）
   - 板块 B 实际找到 Y 条（列出标题，0 条则注明）
   - 板块 C 实际找到 Z 条（列出标题，0 条则注明）
   - 【新增】主题搜索捞到的、名单外的新公司（如有，列出公司名 + 一句话）——供 jacob 迭代名单
   - 【新增】若 A 或 B 写了 🈚️，注明"已跑完穷尽检查"及实际跑过的搜索条数
   - 因去重而丢弃的条目数量及其标题（如有）
   - 文件路径
   - 确认已直推 main
```

## 五、预期效果

- **主题驱动搜索**是最大的防漏杠杆：本周实测中，阶跃星辰、Rime、Reo.Dev、Sprinklr、DriveCentric 五家都不在旧名单里，主题搜索能把它们捞出来
- **handle 修正**让 X 这条线从"每天 11 次空枪"变成真的能搜到东西
- **穷尽检查**把 🈚️ 从"随口一句"变成"有门槛的结论"，直接压低误报空板块的概率
- **报告回流新公司**让名单每天自我迭代，不再需要三个月一次的大审计

## 六、维护建议

在名单标注 `（已核准 2026-07-18）`，**每季度复核一次 handle 与厂商活跃度**——这次 11/22 handle 出错，就是 4 月名单跑到 7 月的自然衰变。
