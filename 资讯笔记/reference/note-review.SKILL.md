# note-review · 笔记内容评审

把"我写完一篇笔记，但不知道好不好、能不能发、要怎么改"这件事，变成一份结构化、可操作的评审报告。

## 一、何时用这个 skill

- 用户粘贴一段中文文章/笔记/草稿，要意见、要改稿、要打分。
- 用户给一个文件路径（`.md` / `.txt` / `.docx`），让你读完再评。
- 上一个 skill（如 `kazike-tuwen`、`cong-hang-dao-la`）刚产出一篇正文，需要审一遍才好发。

不在范围内：直接重写整篇（交给 `kazike-tuwen`）、起标题/写开篇词（交给 `kazike-headline`）、做完整的事实联网核查（独立流程，由用户主动触发）。

## 二、主流程（按步骤跑，不要跳步）

### Step 1 — 路由：搞清楚要审什么、怎么拿到内容

判断输入模式：

- **模式 A（粘贴）**：当前消息含大段中文正文 → 直接拿来用。
- **模式 B（文件）**：当前消息含文件路径或文件名（`.md` / `.txt` / `.docx` 等） → 用 Read 工具读取。`.docx` 用 `python-docx` 提取文本。
- **模式 C（串调）**：上下文里 `kazike-tuwen` / `cong-hang-dao-la` 等 skill 刚产出过正文 → 直接拿那段正文做输入。

如果三种都不明确，问一句：

> 要审的内容在哪里？你可以：把正文粘过来 / 给我文件路径 / 用刚才写好的那篇

不要自己脑补内容。

### Step 2 — 先问两件事（除非已经明确）

用 AskUserQuestion 一次性问两个，不要拆成两轮：

1. **笔记类型**（决定权重表）：
   - 公众号/小红书文章草稿
   - AI 模型/产品测评
   - 选题/灵感笔记
   - 学习/读书/研究笔记

2. **输出粒度**：
   - 全套（默认）：总评 + 评分卡 + Top 问题
   - 只要 Top 问题（快速版）
   - 全套 + 逐段批注（深度版，慢，输出长）

如果用户明显已经在上下文里说过类型（例如刚用 `cong-hang-dao-la` 写完一篇 → 默认是"AI 模型/产品测评"），就不要再问类型，直接默认。

### Step 3 — 加载维度判断标准和权重

根据笔记类型 Read 对应文件：

- 维度细则：`references/dimensions.md`
- 权重表：`references/weights-by-type.md`
- 输出模板：`references/output-templates.md`

不要凭记忆评分，每次都打开 `dimensions.md` 看一眼锚点和反例，再下笔。

### Step 4 — 做一次差异化联网搜索（除非用户说"跳过联网"）

从笔记里提取 2-3 个核心主题词（例如"GPT-5 编程"、"DeepSeek V3 中文"、"AI 副业 小白"）。用 WebSearch 跑一次，搜索词模板：

```
<主题词> 最近 30 天
```

或：

```
<主题词> 2026
```

目的不是核查事实，是看**同主题的近期文章**有没有，找差异化空间。

输出形态：在评审报告的"传播性"维度下加一句：

> 同期已有 X 篇类似主题的文章（如《...》、《...》），建议在 [Y 角度] 上做差异化。

**降级规则**：搜不到 / 搜索失败 / 用户说跳过 → 改用常识判断"这个主题大概率有多少人在写"，并标明"未联网核查"。skill 不能因为搜索失败而崩。

### Step 5 — 切语义段（仅在用户要"逐段批注"时）

不按 `\n\n` 死切。读完全文，识别**语义边界**：一个完整论点、一个完整场景、一个完整段子、一个完整案例 = 一个语义段。

经验值：

- 平均长度 80-200 字
- 单个自然段超过 250 字 → 拆成两段
- 多个短自然段（< 60 字）讲同一件事 → 合并为一段

给每个语义段编号 §1, §2, ..., 后续批注按编号引用。

### Step 6 — 按六维度评分

六个维度：可读性、干货密度、逻辑严密度、事实可信度、风格一致性、传播性。

每个维度独立打分（0-10 整数），写一句具体评价（不要套话）。**评价必须引用原文片段或具体段落编号**，不许写"整体不错"这种空话。

档位语义（必须遵守）：

| 分数 | 含义 |
|------|------|
| 9-10 | 优秀，可截图当范本 |
| 7-8 | 合格，能发 |
| 5-6 | 有硬伤，要改 |
| 3-4 | 基础不达标，建议重写 |
| 0-2 | 不要发 |

加权总分 = Σ(维度分 × 类型权重)，权重表见 `references/weights-by-type.md`。

### Step 7 — 强制反向提问（这步不能省）

评完六维度后，**强制问自己一句**：

> 如果这篇发出去后被骂，最可能的骂点是什么？

把答案并入 Top 问题清单。这一步是为了对抗 LLM 默认偏向夸奖的倾向。如果实在想不出骂点，写"潜在风险：xxx"，不允许跳过。

### Step 8 — 时效检验

扫一遍原文，把容易过时的表述列出来：

- 时间锚词："最近"、"刚发布"、"上周"、"今年"
- 具体版本号、价格、榜单排名
- "目前最强"、"还没人做"、"业界第一"等绝对化表述

输出："半年后再读这篇，[这些点] 可能会变成过期信息，建议加日期或改用相对表述。"

### Step 9 — 写报告

按 `references/output-templates.md` 套模板。三层结构：

1. **一句话总评**（≤ 50 字）：能发 / 有条件能发 / 不能发 + 最大优点 + 最大问题
2. **分维度评分卡**（六维 × 评分 × 一句话评价 + 加权总分 + 档位结论）
3. **Top 3-5 问题清单**：每条三件事——问题在哪（引原文，≤ 30 字）、为什么是问题、怎么改（1-2 句改写参考）

如果用户选了"逐段批注" → 末尾追加第四层：按语义段 §N 顺序，给出每段的批注。

如果做了联网搜索 → 在传播性维度或 Top 问题里带上"差异化"那一条。

### Step 10 — 给 Next Step 建议

报告结尾给 1-2 个后续动作建议（不自动执行），常见的：

- 「要不要让 `kazike-tuwen` 按这些 Top 问题修一遍？」
- 「想就 [某个具体段落] 单独再深入聊聊吗？」
- 「需要我帮你重起一个开头/结尾吗？」

让用户决定下一步，不要自己跳过去执行。

## 三、几条贯穿始终的原则

**评分要有锚点，不要靠感觉。** 每给一个分，都得能在 `dimensions.md` 里指出对应的反例位置。9-10 分必须特别克制，不要轻易给。

**指出问题要引原文。** "第二段太水"是无效批评。"第二段（'今天我们来聊一聊...'）整段都在过渡，没有信息增量，建议删掉直接进案例" 才是有效批评。

**改写建议给一句就够。** 不要写大段重写。给方向和一句示范，让用户/下一个 skill 去落地。

**默认偏严格，不偏宽松。** LLM 评审有偏夸的惯性，要主动对抗。Step 7 的反向提问是这一原则的兜底。

**不要在评审里夹带教育用户。** 用户写完一篇要审稿，不是来听写作课的。直接说哪里能改、怎么改。

## 四、和其他 skill 的衔接

- **上游**：`kazike-tuwen`、`cong-hang-dao-la` 出稿后 → `note-review` 接审
- **下游**：评审完用户决定改稿 → 把 Top 问题清单递给 `kazike-tuwen` 做局部修改
- **平行**：和 `kazike-headline` 不抢活——`note-review` 不出标题/开篇词，标题问题在传播性维度里点一下就行，让用户去 `kazike-headline` 重做

## 五、引用关系（按需 Read）

- `references/dimensions.md` — 六大维度的判断标准、评分锚点、反例对照
- `references/weights-by-type.md` — 四种笔记类型的权重表 + 类型识别启发式
- `references/output-templates.md` — 三层输出（含可选第四层逐段批注）的固定模板
- `references/examples.md` — 一个完整评审示例（输入 → 输出），用作 few-shot 锚点

每次评审至少 Read `dimensions.md` 和 `output-templates.md`。`weights-by-type.md` 在确认笔记类型后 Read。`examples.md` 在不确定输出风格或第一次评审时 Read。
